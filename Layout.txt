Module 1 — App Bootstrap Module

Nome sugerido: app / main

🎯 Responsabilidade

Ponto de entrada da aplicação frontend.

✅ Contém no código:

Inicialização da aplicação

Configuração global:

URL base da API

Rotas principais

Carregamento inicial

Verificação inicial de autenticação

❌ Não contém:

Lógica de login

Lógica de arquivos

Componentes visuais complexos

🧠 Lógica importante

Ao iniciar:

Verifica se existe token

Se existir → valida com backend

Se inválido → limpa sessão e vai para Login

👉 Copilot NÃO deve:

Colocar lógica de negócio aqui

Fazer chamadas de API específicas

Module 2 — Routing Module

Nome sugerido: router

🎯 Responsabilidade

Controlar navegação e acesso às páginas.

✅ Contém no código:

Definição de rotas:

/login

/explorer

/file/:id

Proteção de rotas (route guards):

Rotas públicas

Rotas privadas

Rotas admin-only (se houver)

🧠 Lógica importante

Se não autenticado → redireciona para /login

Se autenticado → permite navegação

Se token inválido → força logout

❌ Não contém:

Verificação de permissões detalhadas

Chamadas diretas ao backend

👉 Isso evita o erro clássico: tudo misturado no router.

Module 3 — Auth State Module

Nome sugerido: auth_state / session

🎯 Responsabilidade

Gerenciar estado da autenticação no frontend.

✅ Contém no código:

Estado:

isAuthenticated

userId

role (ADMIN / USER)

Métodos:

login()

logout()

restoreSession()

Armazenamento do token

🧠 Lógica importante

Logout:

Remove token

Limpa estado

Redireciona para login

Navegação “voltar” não reativa sessão sem token

❌ Não contém:

UI

Chamadas diretas a endpoints de arquivos

👉 Copilot deve entender:
isso é estado, não interface.

Module 4 — Auth API Service Module

Nome sugerido: auth_api

🎯 Responsabilidade

Comunicação com endpoints de autenticação do backend.

✅ Contém no código:

Funções:

login(credentials)

validateToken()

Tratamento de erros:

Credenciais inválidas

Token expirado

❌ Não contém:

Estado global

Redirecionamento

UI

👉 Um módulo só para API evita duplicação.

Dica: "No auth_state, use o localStorage para persistir o token, mas use um Context ou um Hook customizado para que o resto do app 'reaja' quando o usuário deslogar. Se o auth_api retornar erro, o auth_state deve ser o único responsável por limpar os dados."

Module 10 — Error Handling & Feedback Module

Nome sugerido: feedback / errors

🎯 Responsabilidade

Exibir feedback ao usuário.

✅ Contém no código:

Mensagens de erro

Tratamento de erros HTTP:

401 → logout

403 → mensagem de permissão

500 → erro genérico

❌ Não contém:

Lógica de negócio

Chamadas diretas de API

Dica: "Implemente um Interceptor (se usar Axios) ou uma função de 'Fetch Global'. Se qualquer chamada de API retornar 401, esse módulo deve automaticamente disparar o logout() do Módulo 3. Isso evita que o usuário fique 'preso' em uma página branca tentando ler dados sem permissão."
